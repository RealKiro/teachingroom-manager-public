[中文](./WORK_LOG_2026-05-18.md)

Date: 2026-05-18

## Background

This project was initially modeled on an internal classroom equipment workbook and turns spreadsheet-based equipment records into a maintainable web management system. The real workbook has now been removed from the repository and replaced by `初始化数据表格（虚拟）.xlsx`, which contains no real classroom information.

The goal is not only to replace Excel, but to make classroom information a school-level base data source that supports inspections, reviewed changes, Excel import/export, and integration by other departments.

## Original Requirements

1. Manage all classroom information through the web, supporting phones, desktops, and tablets.
2. Support fast display and conditional filtering.
3. Support roles including administrators and inspectors.
4. Inspectors can submit data changes.
5. Administrators review old/new values before official data is updated.
6. Support Excel export for daily work.
7. Initialize fields from the source Excel and allow future field additions.
8. User count is small, about 10 at most, so the system should stay lightweight and maintainable.
9. Use a clean local web tool style similar to the DNS management project.

## Requirement Adjustments

- Classrooms usually have front and back doors, so `front door plate` and `back door plate` fields were added.
- The wording was changed from orientation to building side, such as `A Building · South Side`.
- Building side remains in data but is not currently a top quick filter.
- Classroom data should serve as base data that other departments can connect to.
- Field management is not exposed in the daily UI; new fields can be maintained in the backend.
- User management is visible only to super administrator `admin`.
- Common fields provide suggestions from existing database values in the change dialog.
- Excel upload creates review requests instead of directly overwriting official data.
- Mobile layout needs to be optimized for on-site inspection.
- Super administrators need audit logs and room update history.

## Implemented Features

### Data Import

- The app imports the source workbook on first startup.
- A total of 92 classrooms from A/B/C buildings were imported.
- The original A/B/C sheets were normalized into one data model.

### Classroom Fields

Core fields include:

```text
Building
Building side
Classroom code
Front door plate
Back door plate
Class or purpose
Existing electronic screen
Board type
Audio type
Install date
Department
Planned screen
Planned board
Planned audio
Inspection note
```

### UI Management

- Desktop uses a table view.
- Mobile and tablet use a card view.
- Top filters support search, building, department, plan, and pending-review state.
- Mobile filters, statistics, and classroom cards were made more compact.

### Roles And Permissions

Public-edition first account:

```text
Super administrator username: admin
Password: supplied by INITIAL_ADMIN_PASSWORD or generated randomly on first startup
```

Notes:

- No fixed password or inspector account is built in.
- When no initial password is supplied, read the one-time password from `data/initial-admin-password.txt`.
- `role=admin` means data review permission.
- Only username `admin` is treated as the super administrator.
- The `admin` account cannot be disabled or downgraded.

### User Management

Super administrators can:

- View users
- Create inspectors or administrators
- Update display names
- Update roles
- Enable or disable users
- Reset passwords
- Delete users
- View submit and review counts

User deletion is soft deletion; historical records remain.

### Audit Logs

Super administrators can view:

- Login and logout
- Excel export
- Excel upload review generation
- Change submission
- Approval or rejection
- User creation, update, password reset, and deletion
- Database backup and restore

Filtering is available by keyword, action type, and actor.

### Review Workflow

- Submitted changes do not update official data directly.
- The app creates pending review requests.
- Administrators compare old/new field values.
- Official data is updated only after approval.
- Rejected changes are retained in history.

### Excel Import And Export

- Export respects current filters.
- Export workbook is split by building.
- Uploaded edited Excel files generate pending review requests.
- Compatible with app-exported one-header workbooks and original A/B/C building workbook format.

### Base Data API

Read-only open APIs were added for other departments or internal systems.

Endpoints:

```text
GET /api/open/meta
GET /api/open/fields
GET /api/open/summary
GET /api/open/classrooms
GET /api/open/classrooms/:id
```

Characteristics:

- Read-only
- Uses `X-API-Token`
- Supports CORS
- Does not bypass the review workflow

### Persistent Login Sessions

- Login sessions are stored in SQLite.
- Added the `user_sessions` table.
- Expired sessions are cleaned on startup and during runtime.
- Avoids the Express `MemoryStore` production warning.

### Database Backup And Restore

- Added a super-administrator-only `Database Backup` page.
- The app creates daily automatic database backups.
- Super administrators can create, download, upload, and enable backups.
- Before enabling a database, the app validates integrity, required tables, and the `admin` account.
- A `before_restore` backup is created before restore.
- The service restarts automatically after restore.

## Technical Approach

- Backend: Node.js + Express
- Database: SQLite
- Excel processing: ExcelJS
- Frontend: plain HTML/CSS/JavaScript
- Deployment: Node.js + systemd or Docker Compose

This approach was chosen because the user count is small, deployment is simple, maintenance cost is low, and SQLite files are easy to back up, migrate, and restore.
